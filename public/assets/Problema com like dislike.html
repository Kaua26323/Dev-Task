Essa linha:

🧠 Explicação detalhada da lógica

👉 Contexto

Você está lidando com um array de feedbacks, onde cada item representa o like/dislike de um usuário para uma task específica.

Como cada usuário pode dar apenas 1 feedback por task, precisamos garantir que, ao dar um novo like ou dislike:

  • Remova o feedback anterior do usuário (se existir).

  • Adicione o novo feedback.

🔍 O que a linha faz

ts
const filtered = prev.filter(fb => fb.user !== session.user?.email);

Explicação passo a passo:

  1. prev é o array atual de feedbacks (FeedbackProps[]), ou seja, todos os likes e dislikes existentes.

  2. filter(...) percorre todos os feedbacks.

  3. fb.user !== session.user?.email mantém apenas os feedbacks que NÃO são do usuário atual.

  4. O resultado (filtered) é um novo array onde o feedback do usuário atual foi removido.

🎯 Finalidade

A ideia é garantir que não haja dois feedbacks para a mesma task do mesmo usuário, ou seja, impedir duplicação.

Depois disso, você adiciona o novo feedback:

ts
setFeedback([
  ...filtered, // feedbacks antigos, exceto o do usuário atual
  {
    like: 1,
    dislike: 0,
    user: session.user.email,
    taskId: id
  }
]);

✅ Resumo

A lógica:

  • Remove qualquer feedback anterior do usuário atual (filter).

  • Adiciona o novo feedback (like ou dislike).

  • Atualiza o estado feedback, o que reativa o useEffect e atualiza a contagem de likes e dislikes na tela.

🧭 O que é o App Router?

O App Router é a nova forma de lidar com rotas no Next.js, introduzida oficialmente a partir da versão 13.

Ele substitui (ou complementa) o antigo sistema baseado em pages/ (conhecido como Pages Router) e traz:

✅ Rotas baseadas em componentes React Server/Client
✅ Layouts aninhados
✅ Streaming de conteúdo com React Server Components
✅ Melhor separação entre servidor e cliente
✅ E várias novas funcionalidades como loading states automáticos, error boundaries, e metadata automática (como <title> e <meta>)

📌 O que são Server Components?

Server Components (ou "Componentes de Servidor") são uma funcionalidade do React (introduzida no React 18) e amplamente adotada no App Router do Next.js. Eles permitem que componentes React sejam renderizados no servidor, sem enviar seu código JavaScript para o cliente.

🚀 Benefícios dos Server Components

  1. Melhor performance: como são renderizados no servidor, não viram JavaScript no navegador, o que diminui o tamanho do bundle.

  2. Segurança: você pode acessar dados sensíveis diretamente (ex: tokens de acesso, senhas, etc), pois o código nunca vai pro cliente.

  3. Carregamento mais rápido: como você pode carregar dados diretamente no componente, evita múltiplos fetchs no cliente.

  4. Renderiza HTML puro no servidor, enviado já pronto para o cliente.

📁 Onde e como usar Server Components?

No App Router (app/ folder do Next.js), todos os arquivos .tsx são Server Components por padrão. Exemplo:

tsx
// app/page.tsx (Server Component por padrão)

export default function HomePage() {
  return <h1>Hello from Server</h1>;
}

Você pode usar async diretamente no componente e fazer chamadas para banco de dados ou APIs.

tsx
// app/page.tsx

async function getData() {
  const res = await fetch("https://api.example.com/posts", { cache: "no-store" });
  return res.json();
}

export default async function Page() {
  const posts = await getData();

  return (
    <div>
      {posts.map(post => (
        <div key={post.id}>{post.title}</div>
      ))}
    </div>
  );
}

🔄 Diferença entre Server e Client Component

CaracterísticaServer ComponentClient Component
Executado noServidorNavegador
Tem acesso a banco de dados / APIs protegidas✅ Sim❌ Não
Pode usar hooks como useState/useEffect❌ Não✅ Sim
Enviado como JS ao cliente❌ Não✅ Sim

🧠 Quando usar cada um?

  • Server Component: para renderizar dados, acessar banco, carregar conteúdo dinâmico sem interação.

  • Client Component: quando você precisa de interatividade, como:

    • useState, useEffect

    • onClick, onChange

    • animações, formulários, etc.

🛠 Como declarar um Client Component?

Você precisa colocar no topo do arquivo:

tsx
'use client';

import { useState } from 'react';

export default function MyButton() {
  const [count, setCount] = useState(0);
  return <button onClick={() => setCount(count + 1)}>Contar: {count}</button>;
}

💡 Dica prática: misturando os dois

Você pode compor um Server Component com Client Components dentro:

tsx
// Server Component
import LikeButton from './LikeButton';

export default async function PostPage() {
  const post = await getPost();

  return (
    <div>
      <h1>{post.title}</h1>
      <LikeButton /> {/* Este é client component */}
    </div>
  );
}

🔐 Segurança

Como Server Components não são enviados para o cliente, você pode:

  • Fazer autenticação com tokens privados.

  • Consultar banco diretamente no componente.

  • Evitar expor lógica sensível.

🧩 Cache e revalidação

Você pode controlar como os dados dos Server Components são armazenados:

  • fetch(url) → usa cache (ISR ou SSG).

  • fetch(url, { cache: 'no-store' }) → SSR (como getServerSideProps).

  • fetch(url, { next: { revalidate: 60 } }) → ISR com revalidação de 60 segundos.

Você pode escrever:

tsx

// Agora faz direto:
export default async function Page() {

🧱 O que é a layout.tsx no Next.js?

No App Router (/app folder), o layout.tsx é um componente de layout compartilhado por todas as rotas dentro da pasta onde ele está.
Ou seja, ele envolve todas as páginas filhas e permite definir estruturas fixas como:

  • Header

  • Sidebar

  • Footer

  • Temas globais

  • Fontes, wrappers, etc.

🗂 Estrutura básica com App Router:

plaintext
/app
  layout.tsx       ← layout principal
  page.tsx         ← página da home
  /dashboard
    layout.tsx     ← layout do dashboard
    page.tsx       ← página do dashboard

✅ Regras do layout.tsx

  • Ele deve exportar uma função chamada Layout como default.

  • Ele recebe uma prop obrigatória children (o conteúdo da rota atual).

  • Ele é executado no servidor por padrão (Server Component).

  • Pode ser aninhado (layouts dentro de layouts).

📌 Exemplo básico de um layout.tsx:

tsx
// app/layout.tsx
import './globals.css';
import { ReactNode } from 'react';

export default function RootLayout({ children }: { children: ReactNode }) {
  return (
    <html lang="en">
      <body>
        <header>Meu Header Fixo</header>
        <main>{children}</main>
        <footer>Meu Footer</footer>
      </body>
    </html>
  );
}

  • Este layout será usado em todas as páginas da aplicação, como app/page.tsx, app/sobre/page.tsx, etc.

🧭 Exemplo com layout dentro de uma rota específica

tsx
// app/dashboard/layout.tsx

      <aside style={{ width: '250px' }}>Sidebar</aside>
      <section style={{ flex: 1 }}>{children}</section>
    </div>
  );
}

Este layout será aplicado apenas às rotas dentro de /dashboard, como:

  • /dashboard

  • /dashboard/settings

  • etc.

🔄 Layouts são persistentes (não recarregam)

Uma das grandes vantagens dos layouts no App Router é que eles não são recriados ao navegar entre páginas filhas.

Por exemplo:

  • Se você estiver no layout do dashboard e mudar de /dashboard para /dashboard/settings, o layout continua o mesmo e não re-renderiza. Isso é ótimo para manter performance e estado (como um menu aberto).

export default function ClientLayout({ children }: { children: React.ReactNode }) {
  const [open, setOpen] = useState(false);
  return (
    <div>
      <button onClick={() => setOpen(!open)}>Abrir</button>
      {open && <nav>Menu</nav>}
      <main>{children}</main>

  );
}

🚀 O que é o loading.tsx no Next.js?

O arquivo loading.tsx é um componente especial do App Router no Next.js que é exibido automaticamente enquanto uma rota está carregando de forma assíncrona.

🧠 Quando o loading.tsx aparece?

Ele entra em ação quando:

  1. Um Server Component está sendo carregado.

  2. Uma rota com fetch, await ou consulta a banco de dados está esperando resposta.

  3. O Next.js precisa fazer prefetch ou hidratar dados do lado do servidor.

🗂 Onde ele fica?

Você cria um arquivo chamado loading.tsx na mesma pasta da rota onde quer mostrar o loading.

Exemplo:

plaintext
/app
  layout.tsx
  page.tsx
  loading.tsx     ← mostra enquanto /page.tsx está carregando

/app/dashboard
  layout.tsx
  page.tsx
  loading.tsx     ← mostra enquanto /dashboard/page.tsx está carregando

✅ Regras importantes

  • Ele não precisa ser importado: o Next.js usa automaticamente.

  • Ele é um Client Component por padrão (usa "use client").

  • Pode conter qualquer animação, spinner, texto, imagem, etc.

  • Não bloqueia o layout — o layout.tsx é mostrado junto com o loading.tsx.

// app/dashboard/loading.tsx
export default function Loading() {

}

Esse componente será exibido automaticamente enquanto a rota /dashboard estiver carregando.

🧩 O que são páginas dinâmicas no App Router?

Páginas dinâmicas são aquelas que mudam de acordo com o parâmetro da URL. Por exemplo:

  • /produto/123

  • /usuario/julia

  • /blog/o-que-e-nextjs

No App Router, você cria essas rotas com colchetes no nome do arquivo ou pasta:

bash
app/produto/[id]/page.tsx

Neste caso, [id] é um parâmetro dinâmico, que você pode acessar dentro do componente.

🔍 Como acessar os parâmetros dinâmicos?

No App Router, você não usa mais useRouter() como no Pages Router. Agora você usa a função:

ts
import { useParams } from 'next/navigation';

ou, em componentes assíncronos (Server Components):

ts
type Props = {
  params: {
    id: string;
  };
};

🧠 Exemplo completo com params

tsx
// app/produto/[id]/page.tsx

interface ProdutoPageProps {
  params: {
    id: string;
  };
}

export default function ProdutoPage({ params }: ProdutoPageProps) {
  return (
    <div>
      <h1>Produto ID: {params.id}</h1>
    </div>
  );
}

Acessando:

bash
/produto/123

👉 Vai renderizar: "Produto ID: 123"

📦 Usando fetch com ID dinâmico

Você pode usar params.id para buscar dados:

tsx
// app/produto/[id]/page.tsx

interface ProdutoPageProps {
  params: {
    id: string;
  };
}

export default async function ProdutoPage({ params }: ProdutoPageProps) {
  const response = await fetch(`https://api.exemplo.com/produtos/${params.id}`);
  const produto = await response.json();

  return (
    <div>
      <h1>{produto.nome}</h1>
      <p>{produto.descricao}</p>
    </div>
  );
}

📄 Como criar slugs dinâmicos (URL amigável)

Você pode usar [slug]:

tsx
// app/blog/[slug]/page.tsx

export default function Post({ params }: { params: { slug: string } }) {
  return <h1>Post: {params.slug}</h1>;
}

URL:

bash
/blog/como-usar-nextjs

🔁 Rota dinâmica aninhada

Você pode ter rotas com múltiplos parâmetros:

bash
/app/dashboard/[userId]/[postId]/page.tsx

tsx
export default function PostPage({ params }: { params: { userId: string; postId: string } }) {
  return (
    <div>
      <p>User: {params.userId}</p>
      <p>Post: {params.postId}</p>
    </div>
  );
}

Acessando:

bash
/dashboard/123/abc

🧭 Como navegar entre rotas dinâmicas?

Use o Link:

tsx
import Link from "next/link";

<Link href="/produto/123">Ver produto</Link>

❗ Se quiser usar useParams() (em Client Component)

tsx
'use client';

import { useParams } from 'next/navigation';

export default function ProdutoClientPage() {
  const params = useParams();
  return <h1>ID do Produto: {params.id}</h1>;
}

useParams() só funciona em Client Components ('use client' no topo do arquivo).

🗂 Pasta not-found.tsx para rotas dinâmicas

Se um ID não existir, você pode mostrar uma página de erro customizada:

tsx
// app/produto/[id]/not-found.tsx
export default function NotFound() {
  return <h1>Produto não encontrado.</h1>;
}

E chamar ela com:

tsx
import { notFound } from 'next/navigation';

if (!produto) {
  notFound();
}

O ChatGPT disse:

Claro! Vamos explorar tudo detalhadamente sobre Grupos de Páginas no App Router do Next.js, um recurso poderoso e muitas vezes subestimado que ajuda a organizar melhor rotas e layouts sem afetar a URL final.

🧩 O que são Grupos de Páginas no Next.js?

Grupos de páginas são pastas organizacionais que não afetam a URL da aplicação.

Você cria um grupo de páginas usando parênteses no nome da pasta:

bash
app/(admin)/dashboard/page.tsx

Apesar da pasta (admin) existir, a URL final será:

bash
/dashboard

Ou seja, o nome da pasta entre parênteses é ignorado na rota, mas ajuda a:

  • Organizar melhor o código

  • Aplicar layouts diferentes por grupo

  • Separar responsabilidades (admin, auth, público, etc.)

✅ Exemplo prático

Estrutura:

pgsql
app/
  (admin)/
    dashboard/
      page.tsx
  (auth)/
    login/
      page.tsx
  layout.tsx

URLs resultantes:

  • (admin)/dashboard → /dashboard

  • (auth)/login → /login

💡 Para que serve isso?

  1. Organização do projeto por contexto:

    • Ex: separar rotas públicas, privadas, admin, etc.

  2. Aplicação de layouts diferentes por grupo:

    • Cada grupo pode ter seu próprio layout.tsx

  3. Código mais limpo e escalável

  4. Evita conflitos entre rotas com propósitos distintos

📐 Aplicando layouts por grupo

Você pode definir um layout apenas para o grupo (admin):

tsx
// app/(admin)/layout.tsx
export default function AdminLayout({ children }: { children: React.ReactNode }) {
  return (
    <div style={{ border: '2px solid red', padding: 20 }}>
      <h1>Área Administrativa</h1>
      {children}
    </div>
  );
}

E outro layout para (auth):

tsx
// app/(auth)/layout.tsx
export default function AuthLayout({ children }: { children: React.ReactNode }) {
  return (
    <div style={{ background: '#f0f0f0', padding: 20 }}>
      <h1>Login</h1>
      {children}
    </div>
  );
}

Assim, cada grupo tem seu visual e lógica específicos sem interferir na URL.

🧠 Quando usar grupos de páginas?

SituaçãoUsar grupo de páginas?
Tem várias rotas administrativas✅ sim
Tem rotas públicas e privadas✅ sim
Precisa aplicar layouts diferentes✅ sim
Quer organizar por contexto (ex: marketing, blog, app)✅ sim
Está criando uma rota simples com uma única página❌ não precisa

📦 Exemplo completo de estrutura com grupos

arduino
app/
├── layout.tsx
├── (public)/
│   └── home/
│       └── page.tsx
├── (auth)/
│   └── login/
│       └── page.tsx
├── (dashboard)/
│   └── layout.tsx
│   └── home/
│       └── page.tsx
│   └── settings/
│       └── page.tsx

URLs geradas:

  • /home

  • /login

  • /home (mas com layout do dashboard)

  • /settings

Mesmo que haja duas pastas com home/, elas pertencem a grupos diferentes e renderizam com layouts distintos.

🔄 Reutilização de layouts com grupos

Você pode compartilhar o layout entre vários grupos:

tsx
app/
├── (marketing)/
│   └── layout.tsx
│   └── landing/
│       └── page.tsx
│   └── about/
│       └── page.tsx

Esse layout se aplica a todas as páginas do grupo (marketing) — sem precisar repetir código.

❗ Importante

  • Grupos não afetam a URL — eles são puramente organizacionais.

  • Você pode ter grupos aninhados.

  • Use grupos para aplicar layouts específicos sem alterar a estrutura pública das URLs.

🧭 Como navegar entre páginas em grupos?

Da mesma forma que nas rotas normais:

tsx
<Link href="/dashboard">Ir para dashboard</Link>
<Link href="/login">Fazer login</Link>

Mesmo que dashboard esteja dentro de (admin), você navega normalmente.
